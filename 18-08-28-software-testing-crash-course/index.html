What is Software Testing?

Testing is the process of executing a program or system with the intent of finding errors. (Myers, 1979)
Software Testing is not
  • ... proving that a program contains no errors
  • ... just checking if it works ¯\_(ツ)_/¯
  • First let me ask you, what do you think is the point of testing
  • Is it like, guaranteeing that a program contains no bugs? Making sure it works?
  • Definition
  • And not this stuff. Important from a mindset perspective
  • Your job is to find and eliminate bugs

Anatomy of a Test Case

A test case consists of at least
  • Pre-Condition
  • Input
  • Expected Result

A test case should always be reproducible.

A test case is successful if it uncovered a bug
  • Lets talk about the basic building block of all tests, the test case
  • Important to know what it looks like
  • Pre-Condition: What needs to be done before executing the test case?
  • Input: What actions to take, what input to make? Officially split up, just to keep in mind
  • Expected Result: Whats coming out of the system.
  • At least, so maybe add a OK/NOK column, or split up input
  • Test Cases again should uncover bugs, and they must be reproducible
  • So pay attention to that precondition stuff

Partitions

Partitions are segments of input data that yield the same result.

Data that belongs to the same partition executes the same code paths (usually) - use one test case per partition!

Partitions for the triangle function?

  • (1, 1, 1), (2, 2, 2), ...
  • (1.5, 2, 1.5), (2.5, 2.5, 3), ...
  • (1, 2, 3), (-2, 2, 3), ...
  • First tool: Partitions
  • What are partitions
  • When thinking about code, same paths
  • So, ideally, we need only one test case for each partition
  • Partitions for the triangle?
  • We want be lazy, additional test cases not as much benefit

Boundaries

States

Depending on its state, a system may behave differently even for identical input.
When testing such an application, not only must the input be considered, but also if the system transitions into the correct states. This is called state-based testing.

  • Input identical, but system behaves differently based on state

When do you stop testing?

Tests can never guarantee that a program is without bugs. You stop testing when your tests are good enough.
Testing is risk based. Test to minimize risk and maximize value, not to uncover all possible bugs.
  • Another mindset thing, that ties in with effectiveness
  • When can you stop testing,
  • and the answer is when you consider your tests good enough.
  • You can never test everything, so its always a question of risk, and addition value

Test Automation

Automated tests mean that you save time on test execution. But:

  • Automated tests require maintenance.
  • Automation is a big up-front investment that will take time to repay itself.
  • Automation seldom reveals new errors but instead avoids regressions.
Better think carefully about what you automate and how!
  • You are familiar with automation, obviously
  • Devs usually dont focus on manual, exploratory tests etc
  • Here's some stuff to keep in mind though
  • Maintenance: Your application evolves, and so must your tests - and automated tests are often harder to maintain than manual ones
  • They are expensive: Automation takes time
  • Common misconception: Automation does not reveal errors, it just avoids regression
  • Automation in all its forms is no silver bullet and you need to be careful

The Ten Commandments of Good Unit Tests

  1. Thou shalt treat your test code like your application code.
  2. Thou shalt keep your tests isolated from each other.
  3. Thou shalt make your tests fail.
  4. Thy tests shall test one thing, and one thing only.
  5. Thou shalt keep your tests simple, for they art living documentation.
  • I give you the Ten Commandments of Good Unit Tests - my own creation, so take it with a grain of salt
  • Just what I consider solid advice, you should be able to come back to this if you feel lost
  • 1: Your test code should be coveted and kept squeaky clean like your application code
  • Test code is not a second class citizen, saying so makes your tests sad
  • 2. Test Isolation, you don't want tests influence each other, because that makes them brittle
  • You don't want tests that fail every now and then
  • 3. But you should make every test fail when you write them
  • Write a test that passes, now change the application. Does the test fail?
  • 4. Each test should be a single test case, concentrate on one thing
  • Otherwise hard to read, and brittle. If your test fails, you want to know whats going on, and not doing that makes this hard
  • 5. As simple as possible.
  • Tests are living documentation, and to get the full benefit of that your docs must be readable, no?
  • Lots to take in, don't worry, there's examples later where I explain what I mean some more

The Ten Commandments of Good Unit Tests

  1. Thy tests shall be fast.
  2. Thou shalt use thy tests to improve thy application code.
  3. Thou shalt run thy tests daily.
  4. Thy tests shall be trustworthy.
  5. Thou shalt think like really hard about what you are testing.
  • Heres some more.
  • 6. Unit tests shall be fast
  • Feedback cycle, no use if you wait for 20 minutes
  • 7. Improve app code
  • Writing tests alone improves code, a bit like a sanity check
  • 8. Run tests daily
  • Avoids tests going to rot. Once your tests start to rot, you can throw them away.
  • 9. Trustworthyness
  • Original purpose of test: reveal bugs. If its untrustworthy, its like boy who cried wolf
  • 10. Think hard
  • Testing is not easy, not just break it, but know how to break it, and that takes time
